# 微服务第二阶段拆分路线

本文档基于当前 `master` 分支真实代码结构整理。当前已经完成第一阶段：

- 原后端以 `backend-service` 注册到 Nacos，端口 `8081`
- 新增 `gateway-service` 注册到 Nacos，端口 `10010`
- 已新增 `file-service`，端口 `10020`
- 已新增 `patient-service`，端口 `10021`
- 已新增 `doctor-service`，端口 `10022`
- 已新增 `registration-service`，端口 `10023`
- 前端和接口工具统一通过网关访问
- 网关路由已经按真实控制器前缀划分业务域

第二阶段目标不是一次性把所有包都复制成独立服务，而是按依赖风险逐步拆分，保证每一步都能启动、注册、经网关访问和回退。

## 2. 推荐拆分顺序

### 第一步：先拆 `file-service`

理由：

- `file` 包主要依赖 MinIO、`file_record` 表和通用返回对象。
- 与挂号、支付、药房、检查等事务链路耦合较低。
- 拆出后只需要把网关中的 `backend-file-domain` 改成 `lb://file-service`。

完成标准：

- `file-service` 能独立启动。
- Nacos 服务列表出现 `file-service`，健康实例数为 1。
- `GET /api/files/**`、上传、预览、下载经 `http://localhost:10010` 能访问。
- 原 `backend-service` 中的 `/api/files/**` 可以暂时保留作为回退，但网关只转发到 `file-service`。

### 第二步：拆 `patient-service`（已完成基础拆分）

风险点：

- 当前患者建档会同步创建 `sys_user` 和默认角色关系。
- 拆分前需要决定：患者服务直接写系统表，还是通过 `auth-service/system-service` 接口创建账号。

建议：

- 短期先保持同库，患者服务可以继续访问 `sys_user`、`sys_user_role`。
- 中期再把账号创建下沉到认证系统域。

### 第三步：拆 `doctor-service`（已完成基础拆分）

风险点：

- 医生建档同样会创建系统账号和角色。
- 排班代码当前在 `registration` 包，但业务上和医生科室强相关。

建议：

- 第一轮只拆 `doctor` 和 `department`。
- `/api/schedules/**` 已跟随 `registration-service` 迁移，不切到 `doctor-service`。

### 第四步：拆 `registration-service`（已完成基础拆分）

风险点最高：

- 挂号依赖 Redis Lua 限流、Redisson 分布式锁、Kafka、本地消息表、号源库存、支付事件。
- 拆分时需要重点验证最终一致性和补偿逻辑。
- 当前服务已迁移 `/api/schedules/**`、`/api/registrations/**`、`/api/queue/**`，端口 `10023`。
- 当前仍保持同库，服务内保留查询必需的 `doctor`、`patient`、`payment` 实体和 Mapper，不暴露这些领域的 Controller。

建议：

- 先保持同库。
- 支付成功/超时事件优先使用 Kafka 或明确的服务间接口，不再依赖同 JVM Spring 事件。
- 快速挂号 `/api/registrations/quick` 涉及 Redis 号源库存键、Kafka、本地消息表和最终一致性，后续需要单独做专项压测和补偿测试。

### 第五步：拆 `payment-service`

风险点：

- 当前支付成功和支付超时会影响挂号、检查检验、药房等多个域。
- 原本同 JVM 内的事件监听在拆分后不可用。

建议：

- 支付服务拆分前，先把支付事件改造成 Kafka 消息。
- 各业务服务订阅支付事件后更新自己的业务状态。

### 第六步：拆 `inspection-service`、`pharmacy-service`、`outpatient-service`、`ai-service`

这几个服务依赖更复杂，建议在文件、患者、医生、挂号、支付拆分经验稳定后再处理。

## 3. 每拆一个服务的固定步骤

1. 新建独立 Spring Boot 模块。
2. 设置独立 `spring.application.name` 和端口。
3. 接入 Nacos Discovery，`server-addr` 固定使用 `127.0.0.1:8848`。
4. 复制该业务域最小必需的 controller、service、mapper、entity、dto、vo。
5. 复制或抽取通用能力，例如 `Result`、`BusinessException`、全局异常处理。
6. 本地启动并确认 Nacos 出现新服务。
7. 修改网关对应领域路由的 `uri`。
8. 通过网关访问接口。
9. 保留原 `backend-service` 回退能力，确认稳定后再考虑移除旧入口。

当前已拆出的服务建议按这个顺序启动：

```bash

./mvnw spring-boot:run

cd /Users/aijiaqi/shixun_2026summer/hou1/file-service
../mvnw spring-boot:run

cd /Users/aijiaqi/shixun_2026summer/hou1/patient-service
../mvnw spring-boot:run

cd /Users/aijiaqi/shixun_2026summer/hou1/doctor-service
../mvnw spring-boot:run

cd /Users/aijiaqi/shixun_2026summer/hou1/registration-service
../mvnw spring-boot:run

cd /Users/aijiaqi/shixun_2026summer/hou1/gateway-service
FILE_SERVICE_URI=lb://file-service \
PATIENT_SERVICE_URI=lb://patient-service \
DOCTOR_SERVICE_URI=lb://doctor-service \
REGISTRATION_SERVICE_URI=lb://registration-service \
../mvnw spring-boot:run
```

Nacos 期望服务：

```text
backend-service
gateway-service
file-service
patient-service
doctor-service
registration-service
```

网关路由切换点：

```yaml
- id: backend-schedule-domain
  uri: ${REGISTRATION_SERVICE_URI:lb://backend-service}
  predicates:
    - Path=/api/schedules/**

- id: backend-registration-domain
  uri: ${REGISTRATION_SERVICE_URI:lb://backend-service}
  predicates:
    - Path=/api/registrations/**,/api/queue/**
```

不设置 `REGISTRATION_SERVICE_URI` 时，排班、挂号、队列接口继续走 `backend-service`，用于回退。

## 5. registration-service 当前验证结果

已验证：

- `registration-service` 能独立编译。
- `registration-service` 能连接 Redis、Kafka、Nacos，并注册为 `registration-service:10023`。
- `GET /actuator/health` 返回 `UP`。
- 直连 `GET /api/schedules` 通过。
- 直连 `GET /api/registrations` 通过。
- 直连 `GET /api/registrations/my` 通过。
- 直连 `GET /api/registrations/{id}` 权限校验通过。
- 直连 `GET /api/queue/doctor/me` 通过。
- 直连 `POST /api/schedules`、`PUT /api/schedules/{id}`、`POST /api/schedules/{id}/close` 通过。
- 临时网关 `10011` 设置 `REGISTRATION_SERVICE_URI=lb://registration-service` 后，`GET /api/schedules` 和 `GET /api/registrations/my` 经网关通过。

暂不作为本轮通过标准：

- `/api/registrations/quick` 抢号链路。它依赖 Redis 号源库存键、Kafka 消费、本地消息表和 DB 库存一致性，需要单独专项验证。
- 支付成功/超时后驱动挂号状态变化。当前代码仍使用同 JVM Spring 事件，拆成微服务后需要改造成 Kafka 或显式接口。
