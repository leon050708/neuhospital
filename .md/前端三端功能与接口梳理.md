# 前端三端功能与接口梳理

## 1. 当前项目建议拆分的三个身份

当前项目前端可以按 3 个身份来搭建：

- 患者端
- 医生端
- 管理端

但是要先注意一个现实情况：

- 后端现在已经接入了 JWT 登录态
- 大多数接口已经要求登录
- 但“按角色做严格权限隔离”还没有完全做完

也就是说：

- 前端可以先按角色分菜单、分路由、分首页
- 但不要默认认为后端已经把每个接口都按角色卡死


## 2. 登录注册现状

## 2.1 登录接口

接口：

```text
POST /api/auth/login
```

请求体：

```json
{
  "username": "doctor001",
  "password": "password123"
}
```

返回值里前端最重要的字段：

- `accessToken`
- `refreshToken`
- `role`
- `userType`
- `bizId`

说明：

- 前端登录成功后，应该先保存 token
- 再立即调用 `/api/auth/me`
- 根据 `role / userType / bizId` 决定跳转哪个端


## 2.2 当前用户接口

接口：

```text
GET /api/auth/me
```

请求头：

```text
Authorization: Bearer {accessToken}
```

返回示例：

```json
{
  "userId": 1,
  "username": "doctor001",
  "role": "DOCTOR",
  "userType": "DOCTOR",
  "bizId": 30001
}
```


## 2.3 注册接口

接口：

```text
POST /api/auth/register
```

当前只支持：

- 患者注册

请求体：

```json
{
  "password": "123456",
  "realName": "张三",
  "phone": "13800138000",
  "idCard": "210102199901011234",
  "gender": "MALE"
}
```

注意：

- 当前注册请求没有 `username`
- 后端会直接把 `phone` 当成登录账号
- 所以患者端登录页如果写“用户名”，实际应该提示“手机号”


## 2.4 当前登录注册的前后端出入

当前有几个明显需要注意的点：

1. 患者注册后，登录名不是自定义用户名，而是手机号
2. 医生和管理员没有公开注册接口
3. 前端不要做“医生注册”“管理员注册”页面
4. 当前安全配置里没有放行 `/api/auth/register`
5. 所以注册接口可能会被鉴权拦住，需要后端继续调整


## 3. 三端整体功能划分

## 3.1 患者端功能

建议患者端先做这些页面和功能：

- 登录
- 注册
- 个人档案
- 科室列表
- 医生列表
- 排班列表
- 快速挂号
- 我的挂号
- 取消挂号
- 签到报到
- 待支付项目
- 支付
- AI 问诊


## 3.2 医生端功能

建议医生端先做这些页面和功能：

- 登录
- 候诊队列
- 叫号
- 过号
- 完成接诊
- 门诊病历
- 诊断录入
- 检查申请
- 检验申请
- 检查结果录入
- 检验结果录入
- 处方开立
- CT 文件/目录上传
- CT 分析任务发起
- CT 结果查看

说明：

- 你当前说“医生 CT 部分已经完成”，建议继续沿用现在的 CT 闭环
- 即：目录上传 -> 返回 `file_record.id` -> 创建 CT 任务 -> 轮询结果


## 3.3 管理端功能

建议管理端先做这些页面和功能：

- 科室管理
- 医生管理
- 排班管理
- 患者管理
- 挂号总览
- 药品管理
- 库存调整
- 文件管理


## 4. 患者端业务流程

建议患者端主流程如下：

1. 注册或登录
2. 调用 `/api/auth/me` 获取当前身份
3. 查看科室列表
4. 查看某科室医生列表
5. 查看医生排班
6. 发起挂号
7. 查看我的挂号
8. 查看待支付项目
9. 支付
10. 到院签到
11. 如有需要，进入 AI 问诊


## 5. 医生端业务流程

建议医生端主流程如下：

1. 登录
2. 调用 `/api/auth/me` 获取当前医生业务 ID
3. 查看“我的候诊队列”
4. 叫号接诊
5. 创建或编辑病历
6. 保存诊断
7. 根据需要开检查 / 检验 / 处方
8. 如果是 CT 业务：
9. 上传 CT 目录
10. 创建 CT 分析任务
11. 轮询分析结果
12. 查看分类结果


## 6. 管理端业务流程

建议管理端主流程如下：

1. 登录
2. 维护科室
3. 维护医生
4. 维护排班
5. 查看患者档案
6. 查看全院挂号情况
7. 管理药品与库存
8. 查看文件记录与上传结果

## 10. CT 前端建议走法

当前 CT 前端最推荐走法如下：

1. 前端选择一个 CT 目录
2. 调用 `/api/files/upload-directory`
3. 上传成功后拿到返回的 `file_record.id`
4. 用这个 `id` 作为 `ctImageFileId`
5. 调用 `/api/ct-analysis/tasks`
6. 拿到 `taskId`
7. 前端轮询 `/api/ct-analysis/results/{taskId}`
8. 成功后展示分类结果

创建任务请求体：

```json
{
  "ctImageFileId": 123,
  "analysisType": "B1_B2_CLASSIFICATION"
}
```


## 11. 前端当前最容易踩坑的地方

当前前端接入时最容易出问题的点如下：

### 11.1 注册接口与登录页文案不一致

- 注册时没有 `username`
- 登录时却要求传 `username`
- 实际上患者登录名就是手机号

建议前端：

- 患者端把登录输入框文案写成“手机号”
- 提交时仍然按 `username` 字段传给后端

例如：

```json
{
  "username": "13800138000",
  "password": "123456"
}
```

### 11.2 医生与管理员不能走公开注册

- 医生账号目前应由数据库初始化或管理端创建
- 管理员账号目前也应由数据库初始化或管理端创建

建议前端：

- 不要做医生注册页
- 不要做管理员注册页

### 11.3 业务身份传递还没有完全统一

现在有些接口：

- 会从 token 中识别当前用户
- 有些接口还在要求手动传 `patientId`
- 有些接口还在用 `X-User-Id`

建议前端：

- 统一封装 API 层
- 页面不要直接拼这些字段
- 把这些兼容细节收敛到 `api/*.ts`

### 11.4 后端多数接口只做了“登录校验”，没做严格角色校验

所以现在前端应该：

- 通过 `role / userType` 做页面隔离
- 但联调时不要误以为后端已经完整做了 RBAC

## 13. 当前最建议优先修的后端问题

如果前端要继续稳定搭建，建议后端优先处理下面几项：

1. 放行 `/api/auth/register`
2. 明确“患者注册后手机号就是账号”
3. 逐步取消业务接口里的 `X-User-Id` 兼容写法
4. 用 token 中的 `userId / bizId / userType` 统一身份来源
5. 后续给关键接口补上真正的角色权限控制


## 14. 前端当前最稳的开发顺序

建议开发顺序：

1. 先完成统一登录态管理
2. 登录后统一调 `/api/auth/me`
3. 完成患者端挂号与支付主流程
4. 完成医生端队列、病历、CT 主流程
5. 最后补管理端科室、医生、排班、药品管理

这样最容易先把主业务链路跑通。
