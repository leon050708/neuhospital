## 8. registration-service 测试结果

本轮将排班、挂号、队列一起拆入 `registration-service`，因为 `/api/schedules/**` 的代码在 `registration` 包内，并且与挂号号源库存强相关。

保留内容：

- `registration` 控制器、服务、Mapper、实体、DTO、VO、MQ、Listener
- `common` 返回、异常、事件
- 查询所需的 `doctor`、`patient`、`payment` 实体和 Mapper
- `RedissonConfig`
- `lua/rate_limit.lua`
- 网关头认证相关安全类

没有暴露：

- 医生 Controller/Service
- 患者 Controller/Service
- 支付 Controller/Service

测试数据：

- 创建过一条测试排班，id 为 `29`。
- 测试排班最终已关闭，状态为 `CLOSED`。
- 测试路径使用过演示患者 `9301`、演示医生 `9201/9202`、演示科室 `9101/9102`。

已知问题：

- `GET /api/schedules?pageNo=1&pageSize=2` 返回记录数多于 `pageSize`，并且 `total=0`。这属于原业务分页实现问题，不是服务拆分导致。
- `GET /api/registrations` 曾出现响应较慢或卡住，所以烟测脚本改用 `/api/registrations/my`，并给 curl 加了 `--max-time 25`。
- 运行期间出现过 Hikari 连接校验警告：`This connection has been closed`。接口最终可恢复，但说明远程数据库连接池参数后续需要优化。

未测内容：

- `/api/registrations/quick` 抢号链路未作为本轮通过标准。
- 支付成功后挂号状态自动变更未测。
- 支付超时后挂号取消和库存回补未测。
- Kafka 消息失败重试、重复消费、补偿任务未做专项压测。
- Redis 号源库存初始化和库存不一致修复未测。

不测原因：

- 快速挂号不是普通 CRUD，它涉及 Redis Lua 限流、Redisson 锁、Redis 库存预扣减、本地消息表、Kafka 消费、数据库库存扣减、支付状态联动。
- 当前支付事件仍是同 JVM Spring 事件，拆成独立服务后需要改成 Kafka 或显式接口，否则跨服务不会自然触发。

## 10. 总体结论

当前已经验证通过的微服务链路：

```text
gateway-service
  -> backend-service      认证、系统、回退接口
  -> patient-service      患者详情
  -> file-service         文件上传和查询
  -> doctor-service       医生科室查询和主要管理接口
  -> registration-service 排班、挂号查询、队列查询、排班管理主干接口
```

当前微服务架构的基础连通性已经成立：

- 服务能独立启动。
- 服务能注册到 Nacos。
- 网关能通过 `lb://service-name` 找到服务。
- JWT 能在网关校验并转换为下游 `X-User-*` 请求头。
- 已拆服务的主干读写接口大部分通过。

后续优先处理：

1. 给 `registration-service` 做快速挂号专项测试。
2. 将支付成功/超时事件从同 JVM Spring 事件改为 Kafka 或显式服务间接口。
3. 修复分页 `total=0` 但存在 records 的问题。
4. 排查医生更新接口的 `JDBC rollback failed`。
5. 优化远程 PostgreSQL 的 Hikari 连接池参数，减少连接校验警告。
